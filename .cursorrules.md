## ESTRUTURA PADRÃO DE PÁGINAS (OBRIGATÓRIA)

**Todas as páginas do sistema devem seguir esta estrutura. Não usar o componente Header como container abaixo dos títulos.**

### Hierarquia
```
TopBar (breadcrumbs)
main > PageContainer
  └─ div.py-6.space-y-6
      ├─ PageTitle (ícone + título + rightContent opcional)
      ├─ [Abas, se houver] — ver seção "Design de Abas"
      └─ Conteúdo (cards, tabelas, etc.)
```

### Regras
- **Sem Header**: Não usar `Header` abaixo do TopBar. O título fica dentro do conteúdo, via `PageTitle`.
- **PageTitle**: Componente em `/components/ui/PageTitle.tsx` — recebe `icon`, `title` e `rightContent` (opcional). Use para filtros, botões de ação (ex: Nova Proposta, Novo Cliente).
- **PageContainer**: Envolver todo o conteúdo da página em `PageContainer` para padding consistente.
- **Espaçamento**: Usar `py-6 space-y-6` no wrapper interno para título, abas e conteúdo.

### Exemplos de rightContent
- Dashboard: `PeriodFilter`
- Negócios: botão "Nova Proposta" ou ações (Voltar, Salvar Rascunho, Finalizar)
- Clientes: botão "Novo Cliente"

## DESIGN DE ABAS E BREADCRUMB (Páginas com Sub-navegação)

**Use este padrão quando a página tiver abas internas (ex: Financeiro, Configurações).**

### Design das Abas
- **Título da página**: Fora de qualquer container, direto no conteúdo, com ícone à esquerda (ex: ícone + "Financeiro").
- **Abas**: Texto simples, sem ícones nas abas, sem container/card ao redor.
- **Linha base**: Uma linha cinza (`border-gray-200 dark:border-gray-700`) ocupa toda a largura disponível abaixo das abas.
- **Aba ativa**: Linha azul (`bg-blue-600 dark:bg-blue-400`) apenas sob a aba selecionada, sobrepondo a linha cinza. Usar `-mb-px` no nav para alinhar.
- **Sem separador**: Não há linha ou container entre as abas e o conteúdo abaixo; o conteúdo em cards começa logo após.

### Breadcrumb Dinâmica por Aba
- A breadcrumb deve refletir a aba ativa: `Início > ERP > [Página] > [Nome da Aba]`.
- Exemplo Financeiro: `Início > ERP > Financeiro > Painel` (ou `Performance`, `Contas a Receber`).
- **Implementação**: Elevar o estado da aba para a página; passar `activeTab` e `onTabChange` ao componente de conteúdo; montar o array de breadcrumbs com base em `activeTab`.

## SKELETON LOADING (Carregamento com Placeholder)

**Use skeleton em vez de spinner quando o conteúdo carregado tiver layout complexo, para evitar quebra visual.**

### Quando usar
- Formulários ou páginas que buscam dados do banco e exibem muitos campos.
- Evita o "salto" visual quando o conteúdo substitui um spinner centralizado.

### Padrão
- **Estrutura**: O skeleton deve replicar o layout do conteúdo final (título, campos, botões).
- **Estilo**: `bg-gray-200 dark:bg-gray-700 rounded animate-pulse` em cada bloco placeholder.
- **Dimensões**: Usar alturas e larguras aproximadas dos elementos reais (ex: `h-10` para inputs, `h-4` para labels).
- **Grid/Layout**: Manter o mesmo `grid`, `gap` e `space-y` do conteúdo carregado.

### Exemplo (EmpresaTab)
```tsx
if (isLoading) {
  return (
    <div className="space-y-6">
      <div className="h-7 w-48 bg-gray-200 dark:bg-gray-700 rounded animate-pulse" />
      <div className="grid grid-cols-1 lg:grid-cols-2 gap-6">
        {[1,2,3].map(i => (
          <div key={i}>
            <div className="h-4 w-24 bg-gray-200 dark:bg-gray-700 rounded animate-pulse" />
            <div className="h-10 w-full bg-gray-200 dark:bg-gray-700 rounded-lg animate-pulse" />
          </div>
        ))}
      </div>
    </div>
  );
}
```

## PADRÃO DE DRAWER LATERAL (CRM - Novo Pipeline)

**Referência aplicada em `CrmFunnelContent.tsx` para o drawer "Novo pipeline".**

### Objetivo visual
- O drawer deve abrir pela direita, ocupando a altura completa da viewport.
- Apenas a área do drawer fica iluminada; o restante da tela deve escurecer.
- Não pode haver "faixa sem overlay" no topo nem recorte visual.
- Bordas arredondadas apenas no lado interno do drawer (lado esquerdo).

### Estrutura técnica obrigatória
- Renderizar o drawer via `createPortal(..., document.body)`.
- Usar container raiz `fixed inset-0` com `z-index` alto (ex: `z-[120]`).
- Overlay clicável ocupando toda a tela (`absolute inset-0 bg-black/45`) para fechamento.
- Painel no lado direito com:
  - `h-screen`
  - `w-full max-w-xl`
  - `rounded-l-2xl rounded-r-none`
  - `overflow-y-auto`

### Animação recomendada (sutil e sofisticada)
- Usar `AnimatePresence` + `motion` (Framer Motion).
- Overlay: fade in/out rápido.
- Drawer: slide horizontal leve + pequena variação de opacidade.
- Curva recomendada: `ease: [0.22, 1, 0.36, 1]`.

### Comportamento de interação
- Clique no overlay fecha o drawer.
- Botão `X` no topo fecha o drawer.
- Conteúdo interno com scroll independente (`overflow-y-auto` no painel).

### Formulário de etapas (pipeline)
- Deve permitir:
  - editar nome da etapa;
  - adicionar etapa;
  - remover etapa;
  - reordenar etapa manualmente.
- Reordenação mínima obrigatória:
  - botão de subir (`ArrowUp`);
  - botão de descer (`ArrowDown`);
  - desabilitar botão no topo/fim da lista.
- A ordem visual da lista define `ordem` salva no banco (`idx + 1`).

## Regra de foco (UI)
- Todos os inputs, selects, dropdowns e campos de edicao devem usar foco com borda/ring azul.
- Nao usar foco laranja/amber em formularios do produto.
- Exemplo padrao Tailwind: `focus:outline-none focus:ring-2 focus:ring-blue-200 dark:focus:ring-blue-500/30 focus:border-blue-500`.

## PADRÃO DE DROPDOWN DE FILTROS (CRM LEADS)
- Em CRM Leads, filtros de seleção (ex: Funil, Etapa, Origem, E-mail preenchido) devem usar o mesmo componente visual do dropdown de Funil: botão com chevron + menu customizado.
- Evitar `<select>` nativo nesses filtros para manter consistência de UX/UI.
